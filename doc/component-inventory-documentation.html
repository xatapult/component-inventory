Introduction

Component-inventory is a Xatapult component that generates a website with information about my electronic components (ICs, transistors, etc.).

GitHub: https://github.com/xatapult/component-inventory (git@github.com:xatapult/component-inventory.git)

Base concepts

Base terminology

Term

Description

Identifier

A string to identify something with. An identifier in component-inventory consists of letters, numbers, hyphens (-) and underscores (_). It must start with a letter of number.

Combined-identifier

For categories, identifiers are combined (because categories can have sub-categories, see Defining category items: ci:categories). Combining identifiers is done using the dot (.) as separator, for instance ICS.LOGIC.74HC.

Item-type

The component-inventory system works with items (see below) of a certain item-type, for instance property, category, component, etc. See Item-types.

Item

An item is the basic building block component-inventory works with. An item is always of a certain item-type (see above).

The most common item is the component.

Item-types

Item-type

Description

property

A property item describes some property of a component. For instance its supply voltage, number of connections, etc.

You must define which properties are mandatory/optional for components in a certain category.

Items of this item-type are defined in the component-inventory specification document.

category

A category item describes a category to divide the components into. For instance ICS, or RELAYS.

Categories can have sub-categories.

Items of this item-type are defined in the component-inventory specification document.

price-range

A price-range item describes a price-range to attach to a component. For instance CHEAP (between €0.01 and €0.50), EXPENSIVE (over €10.00), etc.

Price-ranges were chosen over exact pricing because commercial prices differ and it is not always clear what a component exactly costs.

Items of this item-type are defined in the component-inventory specification document.

package

A package item describes the packaging of a component. For instance DIP16.

Items of this item-type are defined in the component-inventory specification document.

location

A location item describes where the component can (usually) be found in my shed.

Items of this item-type are defined in the component-inventory specification document.

component

A component item describes what this system is all about: an electronic component.

Items of this item-type are defined in their own component specification document. The central component-inventory specification document defines where on disk to search for component information.

Automatic creation of <media> elements

If an item that can have a <media> element has none, an attempt is being made to create one.

  • For packages, files in the designated directory with same name as the package identifier are considered.

  • For components, all files in the directory where the component specification document is stored are considered. Any sub-directories are considered to be additional resource directories.

The usage type (the contents of @usage) is determined as follows:

  • Any image is considered usage type overview.

  • A PDF is considered usage type datasheet.

  • Anything else is considered usage type instruction.

Component-inventory XML documents

Namespace usage

All XML for the component-inventory uses the https://eriksiegel.nl/ns/component-inventory namespace.

This document uses the recommended namespace prefix ci for this.

Component specification documents

A component specification document contains the description of a component item. It can be very minimal. Missing information will be filled in as best as possible (often using the special value #unknown).

           count? = xs:integer \| #many \| #unknown
           category-idrefs? = list of (combined-identifier \| #unknown)
           price-range-idref? = identifier \| #unknown
           package-idref? = identifier \| #unknown
           location-idref? = identifier \| #unknown
           location-box-label? = xs:string
           partly-in-reserve-stock? = xs:boolean
           since? = xs:date \| #unknown
           discontinued? = xs:boolean

The additional data document

The additional data document contains information used to build the final website. It contains:

  • Favorite categories and components.

  • The menu (the same for every page).

  • Contents of the home and about page.

  • Some information to put at the top of the item-type overview pages.

It is fairly simple, using Text/SML to define contents.

Text/SML

Any descriptions and longer pieces of content can be stated in both HTML and SML. The rules are:

  • The child elements of a content/description element that are in the SML namespace (http://www.eriksiegel.nl/ns/sml) are considered to be SML. This will be converted into HTML.

    For more information on SML see its component documentation.

    Tip: Use a <fragment xmlns="http://www.eriksiegel.nl/ns/sml"> element to surround larger chunks of SML, so the namespace is defaulted.

  • Child elements in any other namespace are considered to be HTML and will be put in the HTML namespace.

The macro mechanism

The macro mechanism allows using and defining string macros.

Using macros

A reference to a parameter can be done in two different ways:

  • As {$MACRO}

  • As ${MACRO}

Both will result in the text of the macro. To stop macro expansion, double the first curly brace.

Referencing macros can be done in text nodes and attribute values. If the macro does not exist, an error will be raised.

Watch out: it is possible that the software that uses the macro mechanism limits where macros can be used!

Macro expansion flags

A macro reference can be followed by zero or more flags, separated by colons (:). Assume for instance that the macro called REF contains the value alpha:beta. Then referring to it as ${REF:uc} results in ALPHA:BETA (everything upper-case). Referring to is as ${REF:uc:fns} will result in ALPHA_BETA (everything upper-case, filename-safe).

The following flags are defined:

Flag

Description

cap

Capitalize the macro value (first character upper-case, rest as-is).

compact

Remove all whitespace.

fns

Filename safe expansion: replace all forbidden characters in a filename with underscores (_).

fnsx

Filename safe expansion extra: like the fns flag, additionally replace all whitespace characters with underscores.

lc

All lower-case.

normalize

Normalize the macro value (like the normalize-space() function).

uc

All upper-case.

Defining macros

Macros can be defined using the <macrodefs> element. Where this element can appear depends on the application that uses the macro mechanism and must be defined in its schema. It is always the first child element of some other element (usually for something grouping/container like, for instance sections).

<macrodefs>
  <macrodef>*
</macrodefs>
